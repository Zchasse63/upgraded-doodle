# PushPress SDK Reference (@pushpress/pushpress v1.15.0)

## Overview

The **@pushpress/pushpress** package is a Speakeasy-generated TypeScript SDK for the **PushPress Platform API** — the developer-facing API for PushPress, a gym/studio management platform. It wraps a REST API that exposes customers, plan enrollments, classes, events, appointments, reservations, check-ins, messaging, and webhook subscription management.

The SDK is **early-alpha** — the README explicitly warns that breaking changes may occur. It is generated from PushPress's OpenAPI 3.0.0 spec by Speakeasy (`genVersion: 2.879.13`) and is not hand-written. Almost every file in `src/` carries the header `Code generated by Speakeasy (https://speakeasy.com). DO NOT EDIT.`

- **npm**: <https://www.npmjs.com/package/@pushpress/pushpress/v/1.15.0>
- **GitHub**: <https://github.com/PushPress/pushpress-ts> (repo URL declared in `package.json` `repository.url`; note that as of this writing the repo returns 404 to anonymous viewers — it may be private or moved)
- **License**: MIT (per the README badge)
- **Maintainer / author**: `Speakeasy` (npm `author` field). PushPress owns the package scope.
- **Language**: TypeScript (TS 5.8). Published with both CommonJS and ES Modules. Built with `tshy`.
- **Runtimes**: ECMAScript 2020+; Node.js v18 / v20 LTS, Bun v1+, Deno v1.39+, evergreen browsers. Requires Web Fetch, Web Streams (`ReadableStream`), async iterables. Node.js v20+ is required to run the bundled MCP server.
- **Direct dependencies**: `zod ^3.25.0 || ^4.0.0`, `@modelcontextprotocol/sdk ^1.26.0`.
- **Sub-exports** (via `package.json` `exports`): `./types`, `./models/errors`, `./models/components`, `./models/operations`, `./models/webhooks`, `./core.js`, `./funcs/*` (individual tree-shakable functions).
- **Extras shipped with the package**: an installable MCP server binary (`mcp` bin entry, `bin/mcp-server.js`) exposing every SDK method as an MCP tool.

## Installation & setup

```bash
npm add @pushpress/pushpress
# or pnpm add / bun add / yarn add @pushpress/pushpress
```

Minimal client construction:

```typescript
import { PushPress } from "@pushpress/pushpress";

const pushPress = new PushPress({
  companyId: "<your-pushpress-company-uuid>",
  apiKey: process.env["PUSHPRESS_API_KEY"] ?? "",
});

const appt = await pushPress.appointments.appointmentsGet({ id: "<id>" });
```

## Authentication

### Scheme

The SDK uses a single global security scheme:

| Name | Type | Header sent | Environment fallback |
|---|---|---|---|
| `apiKey` | `apiKey:header` | `API-KEY: <value>` | `PUSHPRESS_API_KEY` |

Source: `src/lib/security.ts` line ~248 — `fieldName: "API-KEY"`, `type: "apiKey:header"`, `value: security?.apiKey ?? env().PUSHPRESS_API_KEY`. The raw key value is sent as the literal header value with no `Bearer` prefix.

The header is added by `resolveGlobalSecurity()` which is called by every operation's function in `src/funcs/`.

The single per-operation security scheme is `keys.create` (POST `/keys`), which takes a bearer token instead:

```typescript
await pushPress.keys.create(
  { bearer: process.env["PUSHPRESS_BEARER"] ?? "" },
  { requestBody: { name: "my-key", description: "my key", expiresAt: 1672531200000 } },
);
```

### API key format

**Not documented in any source surfaced during this research.** The user describes their key as `sk_xxxx:yyyy` (colon-separated). The SDK treats the key as an opaque string and does not parse or validate it before injecting it into the `API-KEY` header. Whatever PushPress issues you, send it through verbatim.

### Server base URLs

Defined in `src/lib/config.ts`:

| Name | URL |
|---|---|
| `production` (default) | `https://api.pushpress.com/v3` |
| `staging` | `https://api.pushpressstage.com/v3` |
| `development` | `https://api.pushpressdev.com/v3` |

Pick via `new PushPress({ server: "staging" })` or override with `serverURL`.

### Company ID header

Most operations send a `company-id: <uuid>` request header (renamed from the `companyId` SDK option by an inbound/outbound `remap$` transform). Operations also accept an inline `companyId` field to override the SDK-level default — useful "When using multitenant API keys, specify the company" (the only quoted explanation in the operation docstrings).

### Rate limits

**Not documented in the SDK or any public PushPress page surfaced during this research.** The SDK does define a `RateLimited` error (`src/models/errors/ratelimited.ts`) corresponding to HTTP 429 and applies it to 8 of 49 methods, so the API does rate-limit, but the SDK does not expose limit values or quota headers. Needs confirmation from PushPress support.

## Core data models

All component types live under `src/models/components/` and are re-exported via `@pushpress/pushpress/models/components`. Every type ships with a Zod schema for runtime validation. IDs throughout the API are **UUID strings**; timestamps are **Unix seconds** unless explicitly labeled milliseconds (the API-key `createdAt` / `expiresAt` / `lastUsedAt` are in ms; everything else — webhook `created`, class `start`/`end`, check-in `timestamp`, reservation `registrationTimestamp` — is in seconds).

## Webhooks

### Receiving + verifying

The SDK exposes `pushPress.validateWebhook({ request, secret })` on the top-level client. It takes:

- `request`: either a Fetch `Request`, or `{ body, headers, url, method }`.
- `secret`: the per-webhook signing secret returned by `manageWebhooks.create` (or `rotateSecret`).

It returns the parsed `RequestBody` union of all known event types. On signature mismatch it throws / returns `Err(WebhookAuthenticationError)`.

Standalone form: `import { validateWebhook } from "@pushpress/pushpress/funcs/validateWebhook.js"`.

### Signature scheme (HMAC-SHA256, custom — NOT Svix/standard-webhooks)

From `src/hooks/webhook-security-custom.ts`:

- **Signature header**: `webhook-signature`
- **Algorithm**: HMAC-SHA256
- **Key**: the UTF-8 bytes of the signing secret
- **Message**: `JSON.stringify(parsedBody.data)` — i.e. the SDK signs **only the `data` sub-object**, not the full request body and not a `<timestamp>.<body>` template. There is no `webhook-timestamp` or `webhook-id` header.
- **Encoding**: lowercase **hex** of the HMAC digest

Verification reads the incoming `webhook-signature` header, recomputes the same HMAC of `JSON.stringify(body.data)`, and string-compares. If either is missing, or they differ, the SDK throws `WebhookAuthenticationError`.

> **Gotcha**: because the signature is over `JSON.stringify(data)` and not the raw HTTP body, you cannot verify by re-hashing the raw bytes — you must `JSON.parse` first, then `JSON.stringify` the `data` field. Whitespace and key-ordering differences in the original request body do not matter, but anything that mutates `data` does. If you use the SDK's `validateWebhook` you don't need to think about this; if you roll your own, mirror this exactly.

### Full event-type list and payload shapes

Every webhook payload has the shape:

```typescript
{
  data: <type-specific>;
  created: number;     // Unix seconds
  event: <literal event name>;
}
```

### Events you can subscribe to but do not have an SDK payload model

`createwebhook.ts` accepts `memberapp.updated` and `reservation.noshowed` as valid event types, but **no `RequestBody` schema exists for them in `src/models/webhooks/`** and `validateWebhook` does not check for them. If you subscribe to these you will likely receive `SDKValidationError("No matching schema found for the given webhook payload")` when running them through `validateWebhook`. Parse those payloads yourself, or skip subscribing for now.

## SDK quirks / gotchas

1. **Alpha status**: the README opens with "🚧 Under Construction 🚧 — This SDK is currently in early alpha and is not yet production-ready". Expect breaking changes.

2. **`createCustomer` only creates leads.** The docstring on `pushPress.customers.create` says: *"Note that this endpoint only supports creating leads at this time"*. There is no SDK method to create paying members or active customers — that happens through PushPress's UI / billing flow.

3. **Check-in field names are inconsistent with other models.** Every other model uses `customerId` / `companyId`, but Check-in models use `customer` / `company` (raw UUID strings under those names). Watch out when wiring these into a unified database.

4. **`Reservation.reservedId` is the calendar-item UUID** (class / event / appointment), not a separate reservation slot. The literal slot is `Reservation.id`.

5. **`Reservation.status: "checked-in"`** is hyphenated, while `EnrollmentStatus: "pendactivation"` is one word. The enum values are not stylistically consistent — copy them exactly from the SDK enums (`Status`, `EnrollmentStatus`) and never hand-write them.

6. **Timestamp unit mixing.**
   - Webhook `created`, class `start`/`end`, event `start`/`end`, appointment `startTimestamp`, check-in `timestamp`, reservation `registrationTimestamp` → **Unix seconds**.
   - API-key `createdAt`, `lastUsedAt`, `expiresAt` → **Unix milliseconds**.
   - Enrollment date fields (`startDate`, `endDate`, `lastCharge`, `nextCharge`, `paidUntil`) → **`YYYY-MM-DD` strings**.
   - `Customer.dob` → **`YYYY-MM-DD`**.

7. **Pagination is page-based with a sentinel.** Default `page = 1`, default `limit = 10`. The SDK auto-detects the last page by checking `data.resultArray.length < limit` — so if `limit` is 100 and an API call returns exactly 100, the SDK will try `page + 1` (and may get an empty result). Iterate with `for await (const page of result)`. The pagination implementation lives inline in each `*List.ts` func; there is no `has_more` / `cursor` field exposed.

8. **`countCheckins` accepts `includeHistoricalCheckins` as a STRING enum (`"true"` / `"false"`), not a boolean.** Don't pass `true` — pass `"true"`.

9. **The webhook signature is over `JSON.stringify(body.data)`, not the raw HTTP body.** This is a non-standard scheme (not Svix's `standard-webhooks` v1). If your HTTP framework re-serializes the body before you can hash it, that's actually fine here as long as `data`'s internal structure is preserved — but it is highly unusual and worth flagging to anyone implementing custom verification.

10. **`signingSecret` is only returned at webhook-creation or rotation time.** `manageWebhooks.get` / `list` do not return it. Store it on first response.

11. **No retry information.** The SDK supports outbound retries via `retryConfig`, but inbound webhook retry policy from PushPress to your endpoint is not documented anywhere in the SDK.

12. **`companyId` header is sent as `company-id`** in the actual HTTP headers (kebab-case). The SDK silently remaps `companyId` → `company-id` via `remap$()`. If you build a request by hand or proxy, use the header name `company-id`.

13. **Auth header is `API-KEY` (uppercase, hyphenated)**, not `Authorization`, not `X-API-Key`, not `Api-Key`. The header name is hardcoded at `src/lib/security.ts:248`. Some HTTP frameworks lowercase headers — that's fine; HTTP header names are case-insensitive.

14. **Empty addresses are empty strings, not `null` or `undefined`.** `Customer.address` is always present with all six fields populated (potentially as `""`). Same for the email body fields in `messages.email.send` — `text` and `html` are both required strings, not optional.

15. **`Customer.account` is a discriminated union by `type`.** A `"linked"` account has a `primaryCustomerId`. A `"primary"` account does not.

16. **Standalone functions return `Result<Value, Error>`, not throw.** If you import from `@pushpress/pushpress/funcs/*`, check `res.ok` before reading `res.value`. The class-based methods (`pushPress.x.y()`) unwrap and throw.

17. **`HTTPClient` constructor is exported from `@pushpress/pushpress/lib/http`**, not from the top-level. Useful for injecting proxies (e.g. undici `ProxyAgent`) or adding hooks (`beforeRequest`, `requestError`, etc.).

18. **`PUSHPRESS_DEBUG=true` reveals the API key in logs.** Don't enable in production.

19. **Repo URL in `package.json` is `https://github.com/PushPress/pushpress-ts.git` but at the time of this research the repo returned 404 to anonymous GitHub viewers.** The README references `docs/sdks/*` paths that are not bundled in the npm tarball. To inspect the same source the SDK runs on, install the package and read `node_modules/@pushpress/pushpress/src/`.
